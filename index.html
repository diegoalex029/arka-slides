¿Qué es ARKA?

Plataforma de comercio electrónico de productos tecnológicos basada en microservicios

ms-inventory

Gestión de productos y stock

:8081

ms-orders

Gestión de órdenes de compra

:8082

ms-notifications

Envío de notificaciones

:8083

Cada servicio tiene

  • Su propia base de datos PostgreSQL
  • Su propio contenedor Docker
  • Sus propias credenciales en Secrets Manager

Se comunican via

  • API REST para peticiones del cliente
  • Apache Kafka para eventos internos
  • Sin llamadas directas entre servicios

Infraestructura como código

Arquitectura hexagonal

Ports & Adapters — el dominio no depende de nada externo

domain/model

Entidades, value objects, interfaces (puertos) — Java puro sin frameworks

domain/usecase

Lógica de negocio — no conoce Spring, ni DB, ni Kafka

driven-adapters

r2dbc-postgresql · async-event-bus

entry-points

reactive-web · async-event-handler

Principios aplicados

  • DIP (Dependency Inversion Principle) — casos de uso dependen de interfaces
  • SRP (Single Responsibility Principle) — cada clase una responsabilidad
  • Repository — abstrae acceso a datos
  • Gateway — abstrae sistemas externos

¿Cómo cambiar la base de datos?

Solo cambia el adaptador — el dominio y los casos de uso no se tocan

Saga coreografiada con Kafka

Los servicios reaccionan a eventos — ninguno llama directamente a otro

Flujo — agregar item

ms-orders guarda item PENDING
↓ stock-reserve-requested
ms-inventory valida stock disponible
↓ stock-reserved / stock-reserve-failed
ms-orders item → RESERVED o FAILED
↓ order-created
ms-notifications guarda notificación

HU1 — Registro de productos

Endpoint

POST /api/products/register

Validación en DTO

  • Nombre, precio, stock y categoría requeridos
  • Precio mayor a cero
  • Stock no negativo

Códigos HTTP

  • 200 — registro exitoso
  • 400 — campo inválido o faltante

Implementación

GlobalExceptionHandler

Centraliza todos los errores — mapea cada excepción a su código HTTP

Persistencia sin JPA — R2DBC

No hay @Entity ni cascade — cada operación es explícita y reactiva

Categorías como enum

SMARTPHONE · LAPTOP · AUDIO · MONITOR · ACCESSORIES

HU2 — Actualización de stock

Dos formas de actualizar

Manual — endpoint

PATCH /api/products/updatestock

No acepta valores negativos — validado en DTO y en dominio

Automática — via Kafka

stock-reserve-requested → RESERVED

stock-release-requested → RELEASED

order-confirmed → DECREASE

Historial — stock_movements

CampoDescripción

new_stockStock después
deltaDiferencia
typeRESERVED · RELEASED · DECREASE

HU4 — Orden de compra

Estados de la orden

DRAFT carrito activo
↓ confirmar / cancelar / abandonar
CONFIRMED orden confirmada
ABANDONED sin items
CANCELED cancelada

Estados del item

PENDING RESERVED FAILED CONFIRMED RELEASED CANCELLED

Endpoints principales

POST /api/orderitem

Agrega item — crea orden DRAFT si no existe

GET /api/orders/{id}

Consulta orden con items y total

POST /api/orders/{id}/confirm

Confirma orden — items RESERVED → CONFIRMED

El total solo suma items en RESERVED o CONFIRMED

HU5 — Modificar orden

Operaciones disponibles

Agregar item existente

Si el producto ya está en la orden — suma la cantidad

Eliminar item

DELETE /api/orders/{id}/items/{productId}

Libera el stock reservado automáticamente

Último item eliminado

La orden pasa a ABANDONED automáticamente

Validaciones en el dominio

Solo órdenes DRAFT

La validación vive en la entidad — no en el handler ni en el use case

Liberación de stock

ms-orders publica stock-release-requested → ms-inventory devuelve el stock y registra movimiento RELEASED

HU8 — Carritos abandonados

No implementado — solución técnica diseñada considerando escalabilidad

¿Por qué no polling?

Polling masivo — descartado

Escanear toda la tabla cada X minutos — costoso y no escalable

SQS con delay — propuesto

Al crear la orden se publica un mensaje con delay de 30 min — solo se verifica esa orden específica

Flujo propuesto

Orden DRAFT publica en SQS con delay
↓ 30 minutos después
ms-orders ¿sigue en DRAFT?
↓ si → abandon() · no → descarta
ms-notifications envía recordatorio
